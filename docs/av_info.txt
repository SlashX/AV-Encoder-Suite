═══════════════════════════════════════════════════════════════

28 fisiere totale:

  TERMUX / LINUX (Bash)
  ─────────────────────
  av_launcher.sh               — Meniu interactiv principal (alege encoder, parametri, lanseaza)
  av_common.sh       — Functii partajate (DV, DJI, HDR10+, triple-layer, progress, audio)
  av_encoder_x265.sh    — Encoder H.265/HEVC (HDR10, HDR10+, DV, DJI, triple-layer)
  av_encoder_x264.sh    — Encoder H.264/AVC (profil auto/manual, DJI)
  av_encoder_av1.sh     — Encoder AV1 (libsvtav1 / libaom-av1, film-grain, HDR10+, DJI)
  av_encoder_dnxhr.sh   — Encoder DNxHR (Avid mezzanine, lossless optic)
  av_encoder_apv.sh     — Encoder APV (Samsung profesional, intra-frame, ffmpeg 8.1)
  av_encoder_prores.sh  — Encoder ProRes (Apple profesional, prores_ks, mov obligatoriu)
  av_encoder_audio.sh      — Re-encode doar audio (video stream copy, rapid)
  av_trimconcat.sh         — Trim & Concat pipeline (v36/v37: trim / concat / pipeline / batch trim)
  av_check.sh            — Analiza completa video + audio, export CSV (standalone)
  av_telemetry.sh               — Telemetrie unificata multi-brand (DJI/GoPro/Sony/Garmin/QuickTime)
  av_telemetry.ps1              — PS1 mirror standalone al av_telemetry.sh (Windows)
  av_extractor_gps.sh           — Import GPS extern (GPX/FIT/KML → CSV basic/full/norm, SRT, GPX, KML)
  av_extractor_gps.ps1          — PS1 mirror standalone al av_extractor_gps.sh (Windows)
  PROFILES (src/profiles/) — Profile pre-definite pentru camere specifice
  ────────────────────────────────────────────────────────────────────────
  profiles/example_profile.conf                          — Exemplu profil .conf documentat cu toate campurile
  profiles/dji_action6/DJI_Action6_Airsoft_Indoor.conf   — CRF 22, medium, indoor
  profiles/dji_action6/DJI_Action6_Airsoft_Outdoor.conf  — CRF 20, medium, outdoor
  profiles/dji_action6/DJI_Action6_Moto_Outdoor.conf     — CRF 20, medium, moto
  profiles/dji_action6/DJI_Action6_Moto_Cinematic.conf   — CRF 18, slow, D-Log M + LUT

  ─────────────────────────────────────────────────────────────

  tools/dovi_parser.sh          — Installer dovi_tool (Termux, compilare Rust)

  ────────────────────

  av_check.ps1           — Analiza completa video + audio, export CSV (standalone)

      Contine toata logica intern, nu depinde de av_common.sh.
      Extractorul DJI este integrat in ps1 (nu e fisier separat ca in Termux).
      av_encode.bat a fost eliminat din v24 — PowerShell acoperă complet
      functionalitatea, cu arrays native, error handling real si features complete.
      Scripturile din src/tools/ sunt standalone — se ruleaza manual, nu sunt
      apelate din encodere. Pot fi rulate oricand pentru instalare/actualizare.

3. SETUP TERMUX (Android)
═══════════════════════════════════════════════════════════════

PASUL 1: Actualizeaza Termux
────────────────────────────
  pkg update && pkg upgrade -y

PASUL 2: Instaleaza FFmpeg
──────────────────────────
  pkg install ffmpeg -y

  Verifica encoderele:
    ffmpeg -encoders | grep 265
    ffmpeg -encoders | grep 264
    ffmpeg -encoders | grep av1
    ffmpeg -h encoder=libx265

  Verifica versiunea:
    ffprobe -version

PASUL 3: Instaleaza Termux API (notificari + wake-lock)
────────────────────────────────────────────────────────
  pkg install termux-api -y

  IMPORTANT: Instaleaza si aplicatia Termux:API din F-Droid.
  Fara ea, termux-wake-lock si termux-notification nu functioneaza.

PASUL 4: Acorda acces la stocare
─────────────────────────────────
  termux-setup-storage

  Accepta permisiunea. Dupa aceea poti accesa:
    /storage/emulated/0/
  care este memoria interna a telefonului.

PASUL 5: Instaleaza ExifTool (optional — doar pentru extractor GPS/DJI)
──────────────────────────────────────────────────────────────────────
  Metoda 1 (recomandata):
    pkg install exiftool -y

  Metoda 2 (daca pkg nu gaseste pachetul):
    pkg install perl -y
    cpan App::ExifTool

  Verifica instalarea:
    exiftool -ver

  NOTA: ExifTool este necesar DOAR pentru optiunea "Export date GPS/DJI"
  din launcher. Encodarea video functioneaza fara el.

PASUL 6: Instaleaza Python3 (optional — doar pentru import GPS extern)
──────────────────────────────────────────────────────────────────────
  pkg install python -y

  Verifica instalarea:
    python3 --version

  NOTA: Python3 este necesar pentru: (a) telemetrie GoPro/Sony/Garmin
  (av_telemetry.sh) si (b) "Import GPS extern" (av_extractor_gps.sh)
  care converteste fisiere GPX/FIT/KML. Nu are dependinte externe —
  foloseste doar module standard Python (xml, csv, struct, math, datetime, re).

PASUL 7: Copiaza scripturile si seteaza permisiuni
───────────────────────────────────────────────────
  Copiaza cele 9 fisiere .sh in home Termux ($HOME) sau in folderul Android Scripts.

  Seteaza permisiuni executare:
    chmod +x av_launcher.sh av_common.sh \
              av_encoder_x265.sh av_encoder_x264.sh \
              av_encoder_av1.sh av_encoder_audio.sh av_check.sh av_telemetry.sh av_extractor_gps.sh

  Verifica permisiunile (rwx la inceput = executabil):
    ls -l *.sh

PASUL 8 (optional): Backup scripturi in folder Android
───────────────────────────────────────────────────────
  mkdir -p /storage/emulated/0/Media/Scripts/

  cp av_launcher.sh av_common.sh \
     av_encoder_x265.sh av_encoder_x264.sh \
     av_encoder_av1.sh av_encoder_audio.sh av_check.sh av_telemetry.sh av_extractor_gps.sh \
     /storage/emulated/0/Media/Scripts/

  chmod +x /storage/emulated/0/Media/Scripts/*.sh

  ls -l /storage/emulated/0/Media/Scripts/

4. SETUP WINDOWS
═══════════════════════════════════════════════════════════════

PASUL 1: Instaleaza FFmpeg
──────────────────────────
  1. Descarca FFmpeg full build de la: https://www.gyan.dev/ffmpeg/builds/
  2. Extrage in C:\ffmpeg\ (sau alt folder)
  3. Adauga C:\ffmpeg\bin in variabila de mediu PATH
  4. Verifica in CMD: ffmpeg -version

PASUL 2: Copiaza scriptul
─────────────────────────
  Copiaza av_encode.ps1 in folderul cu fisierele video.
  Scriptul foloseste folderul curent ca INPUT si creaza subfolder "output" pentru rezultate.

PASUL 3 (optional): Instaleaza ExifTool (doar pentru extractor GPS/DJI)
───────────────────────────────────────────────────────────────────────
  1. Descarca ExifTool de pe: https://exiftool.org/ (Windows Executable)
  2. Redenumeste "exiftool(-k).exe" in "exiftool.exe"
  3. Pune exiftool.exe in ACELASI folder cu av_encode.ps1
     (adica langa fisierele video)
  4. Verifica: exiftool -ver

  NOTA: ExifTool este necesar DOAR pentru optiunea "Export date GPS/DJI".
  Encodarea video functioneaza fara el. Scriptul verifica automat daca
  exiftool.exe exista inainte de a porni extractia.

PASUL 4: Permite executia scripturilor PowerShell
─────────────────────────────────────────────────
  Deschide PowerShell ca Administrator si ruleaza O SINGURA DATA:
    Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned

  Dupa aceasta, poti rula scriptul normal:
    powershell -ExecutionPolicy Bypass -File av_encode.ps1

6. FLUX ENCODARE — CUM FUNCTIONEAZA
═══════════════════════════════════════════════════════════════

  Fluxul este identic pe toate platformele (Termux / PS1).
  Diferenta: pe Termux av_launcher.sh deschide scriptul separat per encoder,
  pe Windows totul e intr-un singur fisier (ps1).

  FLUXUL PRINCIPAL (per fisier video):
  ─────────────────────────────────────
  1. Verifica daca output-ul exista deja (>1MB = skip, <1MB = sterge si reincepe)
  2. Detecteaza track-uri DJI (djmd, dbgi, tmcd, cover) → dialog daca gaseste
  3. Construieste map flags (DJI: loop dinamic per stream; non-DJI: standard)
  4. Detecteaza HDR (color_transfer + HDR10+ side_data + DV codec_tag)
  5. Detecteaza Dolby Vision → dialog: stream copy / re-encode HDR10 / skip
  6. Seteaza CRF auto (bazat pe rezolutie) sau foloseste CRF/VBR custom
  7. Aplica resize daca e selectat (scale width, no upscale, aspect ratio pastrat)
  8. Construieste parametri encoder (x265-params / x264 profile+level / av1-params)
  9. Lanseaza ffmpeg in background cu progress file
  10. Afiseaza progress bar (procent, FPS real, ETA)
  11. La finalizare: logheaza dimensiuni, timp, spatiu salvat

  FLUXUL CHECK VIDEO (analiza fara encodare):
  ────────────────────────────────────────────
  1. Scaneaza folderul Input pentru .mp4/.mov/.mkv
  2. Per fisier: extrage codec, rezolutie, FPS, bitrate, HDR, DV, DJI,
     subtitrari, capitole, attachments, canale audio
  3. Calculeaza estimare dimensiune output pentru x265/x264/AV1
  4. Genereaza recomandare encoder (ex: "libx265 — DV singurul suportat")
  5. Afiseaza totul in terminal + exporta CSV cu 30 campuri

  FLUXUL AUDIO-ONLY ENCODE (av_encoder_audio):
  ──────────────────────────────────────────────
  Re-encodeaza DOAR audio, video ramane intact (stream copy).
  Util cand vrei sa schimbi codec-ul audio fara sa re-encodezi video
  (rapid, pastreaza calitatea video 1:1).

  Fluxul:
  1. Alege container (mp4/mkv/mov)
  2. Alege codec audio (AAC/Opus/FLAC + bitrate sau copy)
  3. Scanare fisiere din Input
  4. Per fisier: detectie surround, build audio params
  5. ffmpeg -c:v copy [audio params] → output
  6. Statistici finale + notificare

  FLUXUL EXPORT GPS/DJI (extractordji):
  ──────────────────────────────────────
  Necesita ExifTool instalat. Scaneaza folderul Input si extrage datele
  GPS/telemetrie din fisierele video DJI (inregistrate cu telecomanda GPS
  Bluetooth sau cu aplicatia DJI Mimo conectata la telefon).

  Meniu cu 7 optiuni:
    1) Standard — GPX + KML (Google Earth) + CSV esential
       (Lat, Lon, Alt, Speed, Track, DateTime)
    2) Full Data — GPX + KML + CSV cu absolut toate metadatele (setari camera,
       firmware, orientare, bitrate — tot ce a salvat camera)
    3) Subtitrare — fisier .SRT pentru VLC (viteza, altitudine,
       coordonate afisate ca subtitrare in timp real)
    4) Totul — GPX + KML + CSV basic + CSV full + SRT
    5) Raw streams — extrage stream-urile DJI ca fisiere separate:
       djmd → _djmd.bin (date GPS/telemetrie raw)
       dbgi → _dbgi.bin (date debug DJI, ~295 MB)
       tmcd → _tmcd.bin (timecode raw)
       cover → _cover.jpg (thumbnail JPEG al inregistrarii)
       Aceasta optiune foloseste ffmpeg (nu ExifTool) si este utila
       ca backup al datelor originale inainte de editare video.
    6) Elimina metadata DJI (remux rapid) — sterge track-uri DJI din sursa
       fara re-encode (ffmpeg -c copy). Sub-dialog:
         1) Elimina doar debug (dbgi ~295 MB) [implicit]
         2) Elimina GPS + debug (djmd + dbgi)
         3) Elimina tot (djmd + dbgi + tmcd + cover)
       Output: _clean.ext in folderul output. Fisierul original ramane intact.
    7) Anulare

  Optiunile 1-4 necesita ExifTool. Optiunile 5-6 necesita ffmpeg.
  Fisierele .fmt (template-uri ExifTool) se genereaza automat si se sterg
  dupa procesare. Nu trebuie descarcate separat.

  FLUXUL IMPORT GPS EXTERN (av_extractor_gps):
  ──────────────────────────────────────────────
  Necesita python3 instalat (pkg install python).
  Converteste fisiere GPS de la orice dispozitiv (Garmin, Huawei, Apple
  Watch, Strava, Komoot, etc.) in formate utile.

  Formate input suportate:
    .GPX — GPS Exchange Format (XML, universal)
    .FIT — Flexible and Interoperable Data Transfer (binar, Garmin/sport)
    .KML — Keyhole Markup Language (Google Earth, Maps)

  Pune fisierele .gpx / .fit / .kml in InputVideos/ si ruleaza scriptul.

  Meniu cu 6 optiuni de export:
    1) CSV esential — Lat, Lon, Alt, Speed(m/s), DateTime
    2) CSV complet — toate metadatele disponibile (inclusiv HR, cadence,
       power, temperatura daca sunt prezente in fisierul sursa)
    3) Subtitrare SRT — speed km/h + altitude + coordonate (+ HR/cadence
       daca fisierul FIT contine date de la senzori)
    4) Totul — CSV basic + CSV full + SRT
    5) GPX & KML — conversie intre formate:
       GPX → KML, FIT → GPX + KML, KML → GPX
    6) Anulare

  GPX: parsat cu Python XML. Speed calculat automat din distanta/timp
  daca nu e prezent in extensions.
  FIT: parser binar pur Python (fara dependinte externe). Citeste record
  messages, converteste semicircles in grade, altitude si speed in unitati
  standard. Extrage si HR, cadence, power, temp daca sunt prezente.

  FLUXUL TRIM & CONCAT (v36/v37 — av_trimconcat.sh / Invoke-*Flow):
  ──────────────────────────────────────────────────────────────
  Submeniu cu 4 flow-uri + back:
    1) Trim clip (un fisier)
    2) Concat clips (unire)
    3) Trim + Concat + Encode (pipeline complet)
    4) Batch trim (v37: aceleasi cuturi pe N fisiere)
    5) Inapoi

  La intrare: scan foldere reziduale din Temp/. Daca exista subfoldere
  trim_*/concat_*/pipeline_*/preview_* (sesiuni anterioare intrerupte sau esuate),
  se afiseaza lista cu size (MB) + age (h/z) si 3 optiuni:
    1) Pastreaza toate — pentru recovery manual
    2) Sterge >24h [default] — curatenie sigura (procese vii protejate)
    3) Sterge toate — curatenie completa

  FLOW 1 — TRIM UN FISIER:
    1. Listare fisiere din InputVideos/ cu durate
    2. Selectie fisier
    3. Loop multi-cut: pentru fiecare segment
       - Introducere start/end (format flexibil: 45 / 1:30 / 1:05:30)
       - Confirmare clip rezultat
       - Alegere precizie:
         1) Stream copy (instant, lossless, ±1-2s la keyframe) [default]
         2) Re-encode (exact, frame-accurate, mai lent)
       - Dialog re-encode: codec (x265/x264) + CRF + audio (copy/aac/eac3)
       - Output: <nume>_trim<N>_<start>.<ext> cu collision auto-increment
    4. Intreaba daca mai vrei alt segment din acelasi fisier

  FLOW 2 — CONCAT CLIPS:
    1. Listare fisiere + selectie cu range syntax:
       all / 1,3,5 / 1-5 / 1-3,7,10-12
    2. Ordine:
       1) Nume alfabetic [default]
       2) Data modificare
       3) Dimensiune
       4) Manual (introdu ordinea: 3,1,2)
       5) Pastreaza ordinea selectiei
    3. Afisare ordine finala cu durate + durata totala
    4. Nume output (default: concat_YYYYMMDD_HHMMSS)
    5. Container: mkv [default] / mp4 / mov
    6. Compat check — ffprobe extrage codec+WxH+fps+pix_fmt pentru fiecare:
       a) Toate identice → optiune stream copy (concat demuxer, instant, lossless)
                         sau re-encode (compresie suplimentara)
       b) Diferente → re-encode OBLIGATORIU via concat filter
    7. Re-encode: codec (x265/x264) + CRF + audio (aac/eac3/copy)
    8. Executie → cleanup temp automat daca OK, pastrat daca esuat

  FLOW 3 — TRIM + CONCAT + ENCODE (PIPELINE):
    Pas 1: Selectie fisiere (range syntax) + sort (ca la Concat)
    Pas 2: Indicare care fisiere au nevoie de trim
           Exemple: none / 1,3 / 1-2 / all
           Pentru fiecare fisier selectat: loop multi-segment (start/end)
    Pas 3: Setari encode GLOBALE (o singura data pt toate):
           - Mod encode (v37): 1) Video+Audio re-encode [default]  2) Audio-only (video copy)
           - Codec (daca nu audio-only): libx265 [default] / libx264 / libsvtav1
           - CRF [default 22]
           - Preset: medium [default] / slow / fast
           - Audio: aac 192k [default] / eac3 224k / copy
    Pas 4: Nume output (default pipeline_YYYYMMDD_HHMMSS) + container (mkv/mp4/mov)
           - Capitole automate (v37): 1) Da [default] 2) Nu — 1 capitol per segment
    Pas 5: Rezumat pre-executie:
           - Lista fisiere cu tag [FULL] sau [TRIM xN]
           - Durata finala estimata
           - Temp estimat (MB) — proportional cu bitrate-ul sursei
           - Setari encode + nume output (sau "AUDIO-ONLY" daca audio_only)
           - Info HDR daca e cazul (auto-detectie inainte de Pass 3)
           - Preview thumbnails (v37, opt-in): 3-frame tile 320p / fisier in preview_<ts>/
           - Confirm (d/n) [default: d]
    Pas 6: Executie in 3 pass-uri explicite:
           [Pass 1/3] Trim stream copy — fiecare segment → subdir temp
                      Fisierele FULL (fara trim) NU se copiaza (referinta directa)
                      Tracking parallel seg_durations[] pt timeline (v37)
           [Pass 2/3] Verificare compat (codec+rez+fps+pix_fmt) pe fisierele generate
                      Daca compat: generare concat.txt (demuxer path)
                      Daca diferite: switch automat la concat filter
                      Audio copy → fallback aac 192k cand se foloseste concat filter
                      audio_only + filter incompat → fallback la full re-encode
                      v37: Smart stream copy detection — daca codec source == codec target,
                      opt-in skip re-encode total (instant `-c copy`)
           Pre-Pass 3 (v37): chapters.txt FFMETADATA1 daca opt-in
           Pre-Pass 3 (v37): HDR mode detection (sdr/hdr10/hdr10plus/hlg/dv/mixed)
                      hdr10/hlg + libx265 → auto inject pix_fmt yuv420p10le + color_* + x265-params
                      hdr10plus → opt-in extragere via hdr10plus_tool, fallback HDR10 la lipsa tool
                      dv → fallback HDR10 + warning ca DV RPU nu se pastreaza
                      mixed → no inject + warning
                      codec ≠ libx265 → no inject + warning
           [Pass 3/3] Concat + Encode intr-un singur ffmpeg, cu progress bar real:
                      smart_copy: -f concat -i concat.txt -c copy (instant, lossless)
                      audio_only: -f concat -i concat.txt -c:v copy + audio re-encode
                      filter:     -i per fisier + filter_complex concat (lent dar robust)
                      demuxer:    -f concat -safe 0 -i concat.txt (rapid, input unificat)
                      Toate primesc condiționat: -i chapters.txt + -map_chapters <idx> (v37)
                      Re-encode primeste hdr_pix_args + hdr_color_args + hdr_x265_args (v37)
                      Comun (re-encode): -c:v CODEC -crf CRF -preset PP -map_metadata 0
    Pas 7: Cleanup temp daca output OK, pastrat daca esuat
    Stats finale: size output, ratio compresie vs input, durata, nr. segmente

  FLOW 4 — BATCH TRIM (v37):
    1. Listare fisiere + multi-select (range syntax)
    2. Calcul min_dur (limita superioara cuturi)
    3. Loop: defineste 1-N cuturi start:end (aplicate la TOATE fisierele)
    4. Mod: 1) Stream copy [default, instant]  2) Re-encode (codec/CRF/audio)
    5. Loop fisiere × cuturi → output `<basename>_btrimN_<start>.<ext>`
       Skip automat fisiere mai scurte decat end_s (cu log explicit)
       Sumar final: OK / FAIL / SKIP
    Util pentru: scoatere intro/outro din toate clipurile dintr-o sedinta cu un singur set de cuts.

  PROGRESS BAR REAL (v37/v38):
  Toate apelurile ffmpeg heavy (Trim re-encode, Concat copy + filter, Pipeline Pass 3
  toate variantele, Batch trim re-encode, ENCODE FLOW PRINCIPAL v38) folosesc
  -progress file watched paralel.
  Afiseaza in terminal: "<LABEL>: NN% | FPS: X.X | ETA: HH:MM:SS"
  Label dinamic (v38) bazat pe encoder activ:
    bash: uppercase ENCODER_NAME (LIBX265, AV1, DNXHR, APV, PRORES, LIBX264)
    PS:   HEVC / H264 / AV1-SVT / AV1-AOM / DNxHR / ProRes / APV / *-NVENC / *-QSV / *-AMF
  La exit code != 0: tail -10 din stderr capturat in fisier temporar (root cause vizibil),
  appended si la LOG_FILE pentru istoric complet.

  SMART STREAM COPY (v38):
  Detectie automata in run_encode_loop / PS encode main: daca codec source == codec target
  (libx265↔hevc, libx264↔h264, av1↔av1) si NU sunt transformari planificate, prompt opt-in:
    "⚡ SMART COPY: source este deja X, identic cu target (libX) (bitrate sursa ~Y kbps)."
    "Re-encode redundant — pierde calitate fara beneficiu real de compresie."
    "Stream copy total in loc de re-encode? (D/n) [default: D]:"
  Skip detectie cand exista: VIDEO_FILTER (scale/vidstab/preset), AUDIO_NORMALIZE,
  LOG profile (Apple/D-Log M/Samsung), HDR10+ side data, Dolby Vision, TRIPLE_LAYER_MODE,
  FPS modification — toate cer re-encode legitim.
  Caz tipic: user descarca .mkv HEVC, alege "encode HEVC CRF 22" — in loc de 4h re-encode
  cu degradare, primeste smart copy instant lossless.

  Termux: /storage/emulated/0/Media/Temp
  Windows: <folder av_encode.ps1>/Temp

  Exclus din git (.gitignore).

7. OPTIUNI DETALIATE
═══════════════════════════════════════════════════════════════

  ENCODER
  ───────
  libx265    — H.265/HEVC. Compresie ~40% mai buna decat H.264.
               Suporta HDR10, HDR10+, Dolby Vision. Recomandat general.
  libx264    — H.264/AVC. Compatibilitate maxima (orice dispozitiv).
               NU suporta HDR nativ (fara metadata MaxCLL/MaxFALL).
               Profil selectabil: high (8bit), high10 (10bit), high422.
  libsvtav1  — AV1 via SVT. Rapid, recomandat pe telefon.
               Compresie ~20-30% mai buna decat HEVC. Suporta film-grain.
  libaom-av1 — AV1 via libaom. Calitate maxima, foarte lent.
               Film-grain via -denoise-noise-level (nu synthesis complet).

  CONTAINER
  ─────────
  mkv   — Flexibil, suporta orice codec/subtitlu/DV/track-uri DJI. Recomandat default.
  mp4   — Compatibil maxim (web, Smart TV, telefoane). Adauga -movflags +faststart.
          Incompatibil cu subtitrari PGS/DVDSUB si track-uri DJI (codec "none").
  mov   — Apple / Final Cut Pro. Aceleasi limitari ca mp4.
  mxf   — Avid native. Folosit exclusiv cu DNxHR.

  ENCODER 4 — DNxHR (Avid, codec video mezzanine)
  ─────────────────────────────────────────────────────────────
  DNxHR este un codec video profesional dezvoltat de Avid, standard in post-productie
  broadcast si film. Spre deosebire de x265/x264/AV1 care sunt codecuri lossy destinate
  distributiei, DNxHR este un codec mezzanine — lossless optic, dimensiune predictibila,
  viteza de decodare maxima, perfect pentru timeline editing.

  CAND SE FOLOSESTE:
    - Trimiti materialul unui editor care lucreaza in Premiere Pro, Resolve sau Avid
    - Vrei un master intermediar fara degradare pentru editare/color grading
    - Sursa este .vob DVD sau .m2ts Blu-ray si vrei un fisier editabil direct
    - Ai nevoie de un codec cu decodare rapida pe timeline (nu CPU-intensive ca x265)

  CAND NU SE FOLOSESTE:
    - Distributie finala (fisierele sunt mari — zeci de GB pentru film intreg)
    - Stocare pe termen lung (foloseste x265 CRF 18-22 pentru arhivare)
    - Streaming sau upload (foloseste x265/x264/AV1)

  PROFILURI DNxHR:
  ──────────────────────────────────────────────────────────────────────
  1) DNxHR LB — ~45 Mbps la 1080p/25fps
     Offline editing cu spatiu limitat. Calitate vizuala acceptabila pentru
     montaj. NU recomandat pentru color grading sau livrare finala.
     pix_fmt: yuv422p10le (10-bit, 4:2:2)

  2) DNxHR SQ — ~145 Mbps la 1080p/25fps  [implicit recomandat]
     Standard Quality — cel mai echilibrat profil. Folosit pentru livrare
     interchange intre editare si color grading. Vizual identic cu sursa
     in conditii normale de vizionare.
     pix_fmt: yuv422p10le (10-bit, 4:2:2)

  3) DNxHR HQ — ~220 Mbps la 1080p/25fps
     High Quality — spatiu cromatic si detaliu superior fata de SQ.
     Recomandat pentru color grading intensiv sau livrari broadcast.
     pix_fmt: yuv422p10le (10-bit, 4:2:2)

  4) DNxHR HQX — ~220 Mbps la 1080p/25fps
     12-bit, singurul profil DNxHR recomandat pentru HDR workflow.
     Pastreaza range-ul HDR corect (10 stop vs 8 stop la profilele 10-bit).
     Foloseste pentru surse HDR10 cand vrei sa editezi/gradezi in HDR.
     pix_fmt: yuv422p12le (12-bit, 4:2:2)

  5) DNxHR 444 — ~440 Mbps la 1080p/25fps
     4:4:4 chroma sampling — zero compresie cromatica. Necesar pentru
     VFX, compositing, keying verde/albastru, lucrari unde fidelitatea
     culorii la nivel de pixel este critica.
     pix_fmt: yuv444p10le (10-bit, 4:4:4)

  DIMENSIUNI ESTIMATE (1080p/25fps, 1 ora material):
    DNxHR LB  : ~20 GB
    DNxHR SQ  : ~65 GB
    DNxHR HQ  : ~99 GB
    DNxHR HQX : ~99 GB
    DNxHR 444 : ~198 GB

  CONTAINERE:
    .mov — QuickTime, compatibil cu Final Cut Pro, Premiere Pro, Resolve
    .mxf — Avid native, compatibil cu Media Composer, Interplay

  LIMITARI:
    - Track-urile DJI (djmd/dbgi/tmcd) nu sunt compatibile cu .mov/.mxf
      si sunt omise automat. Metadatele raman in fisierul original.
    - Nu are CRF, preset de viteza, tune sau VBR — bitrate este fix per profil.
    - Fisierele sunt mult mai mari decat x265/AV1 — plan spatiu in consecinta.
    - Subtitrari PGS/dvd_subtitle incompatibile cu .mov — omise automat.

  SURSE DVD SI BLU-RAY
  ─────────────────────────────────────────────────────────────
  Suita proceseaza direct fisiere .vob (DVD) si .m2ts/.mts (Blu-ray)
  fara tooluri externe, cu conditia ca discul sa NU fie protejat prin
  CSS (DVD) sau AACS/BD+ (Blu-ray comercial).

  Discuri NEPROTEJATE (filmate personal, backup propriu):
    → Copiaza fisierele direct in InputVideos si encodeaza normal.

  Discuri PROTEJATE (comerciale):
    → Necesita MakeMKV sau similar pentru decriptare inainte.
    → Dupa decriptare, .mkv rezultat se proceseaza normal.

  FORMAT .VOB (DVD)
  ─────────────────
  Codec video: MPEG-2 (interlasate in marea majoritate a cazurilor).
  Codec audio: AC3/DTS (surround), PCM.
  Subtitrari:  dvd_subtitle (format imagine, bitmap).
               Incompatibile cu mp4/mov → omise automat (-sn).
               Se pastreaza doar in container mkv.

  ATENTIE DEINTERLACE: Sursele DVD sunt aproape intotdeauna interlasate
  (linii orizontale vizibile in miscare rapida). Activeaza filtrul
  Deinterlace (optiunea 7 din meniu filtre) pentru rezultate corecte.

  DVD-ul fizic imparte titlul in segmente de max 1GB:
    VTS_01_0.VOB = meniu (nu se encodeaza)
    VTS_01_1.VOB, VTS_01_2.VOB, ... = continutul titlului

  Optiuni disponibile in launcher (Termux):
    1) Encodeaza fiecare .vob individual — simplul ca fiecare segment
       devine un fisier encodat separat.
    2) Import titlu DVD complet — specifica calea folderului VIDEO_TS/,
       launcher-ul concateneaza automat segmentele VTS_01_*.VOB intr-un
       singur fisier DVD_titlu_complet.vob in InputVideos, gata de encodat.
       ATENTIE: calea catre VIDEO_TS/ nu trebuie sa contina spatii —
       protocolul concat: al ffmpeg nu suporta spatii in cai de fisiere.

  FORMAT .M2TS / .MTS (Blu-ray)
  ──────────────────────────────
  Codec video: H.264 (AVC) sau H.265 (HEVC), progresiv (de obicei).
               Nu necesita deinterlace.
  Codec audio: DTS-HD, TrueHD, AC3, AAC.
  Subtitrari:  PGS (hdmv_pgs, format imagine bitmap).
               Incompatibile cu mp4/mov → omise automat (-sn).
               Se pastreaza doar in container mkv.

  Fisierele .m2ts de pe Blu-ray sunt fisiere unice per titlu
  (de obicei in BDMV/STREAM/XXXXX.m2ts) — se pun direct in InputVideos.

  REZOLUTIE OUTPUT
  ────────────────
  1) Pastreaza originala [implicit] — fara resize
  2) 3840 — 4K UHD
  3) 2560 — 2K / 1440p
  4) 1920 — Full HD 1080p
  5) 1280 — HD 720p
  6) Custom — introdu width manual (minim 320, numar par)

  Cum functioneaza:
    ffmpeg primeste "-vf scale=WIDTH:-2" care calculeaza height-ul automat
    din aspect ratio-ul sursei. -2 garanteaza height par (necesar h264/h265).
    Nu face upscale: daca sursa e mai mica sau egala cu target-ul ales,
    rezolutia ramane neschimbata.
    La Dolby Vision stream copy: resize-ul NU se aplica (e copy, nu re-encode).

  FRAME RATE (FPS) OUTPUT
  ───────────────────────
  1) Pastreaza original [implicit] — fara conversie FPS
  2) 60 fps
  3) 50 fps
  4) 30 fps
  5) 25 fps (standard PAL/european)
  6) 24 fps (cinematic — film pe pelicola)
  7) 23.976 fps (24000/1001 — standard Blu-ray, Netflix, content US)
  8) Custom — introdu FPS manual (ex: 29.97, 48, 120)

  Dupa alegerea FPS-ului, se alege metoda de conversie:
  1) Drop/duplicate frames [implicit] — rapid. ffmpeg arunca sau duplica
     frame-uri pentru a atinge FPS-ul tinta. Metoda standard.
  2) Motion interpolation — calitate mai buna, genereaza frame-uri
     intermediare prin analiza miscarii. FOARTE lent (10-20x mai lent).
     Pe telefon poate dura ore. Recomandat doar pe PC.

  Nu face FPS upscale: daca sursa e 30fps si alegi 60fps, ramane 30fps
  (nu duplica frame-uri inutil). Conversia se aplica doar downward.
  Audio-ul NU necesita modificari — ffmpeg sincronizeaza automat.
  La Dolby Vision stream copy: FPS-ul NU se aplica (e copy, nu re-encode).

  FILTRE VIDEO (optional)
  ───────────────────────
  Filtrele video se aplica per-fisier, dupa scale si inainte de FPS, toate
  combinate intr-un singur -vf. Nu afecteaza audio.

  1) Fara filtre [implicit]
     Niciun filtru aditional — output identic cu sursa (minus re-encode).
     Recomandat cand materialul este curat si nu are probleme vizibile.

  2) Denoise light — nlmeans h=1.0:s=7:p=3:r=5
     Non-local means denoising, intensitate redusa.
     Pastreaza detalii fine (texturi, fir de par, text).
     Potrivit pt: footage 4K modern cu zgomot usor de ISO ridicat,
     drone footage in conditii bune de lumina, camere de actiune.
     Impact performanta: ridicat (nlmeans e lent pe CPU).

  3) Denoise medium — hqdn3d luma_spatial=4 chroma_spatial=3
     High Quality DeNoise 3D — filtru clasic rapid, eficient.
     Lucreaza pe ambele axe spatiale si temporale simultan.
     Potrivit pt: fisiere cu zgomot moderat, encode-uri lungi unde
     viteza conteaza, material de arhiva digitizat.
     Impact performanta: scazut (hqdn3d e rapid).

  4) Denoise strong — nlmeans h=3.0:s=7:p=5:r=9
     Non-local means agresiv, fereastra mare de cautare (r=9).
     Elimina zgomot ridicat dar poate inmuia detalii fine.
     Potrivit pt: VHS digitizat, surse TV analogice, footage nocturn
     cu ISO foarte ridicat, material cu artefacte de compresie.
     Impact performanta: foarte ridicat (cel mai lent filtru).

  5) Sharpen light — unsharp luma_amount=0.8 chroma_amount=0.4
     Unsharp mask — ascutire clasica, naturala.
     Mareste contrastul local la margini fara a introduce halos.
     Potrivit pt: footage usor neclar din miscare, conversii de
     rezolutie (dupa downscale), material mai vechi.
     Impact performanta: scazut.

  6) Sharpen medium — cas strength=0.6
     Contrast Adaptive Sharpening (AMD CAS).
     Adapteaza intensitatea per-pixel in functie de continut local.
     Evita supraascutirea in zone deja clare.
     Potrivit pt: footage modern, gaming capture, screencasts.
     Impact performanta: foarte scazut (GPU-friendly, eficient pe CPU).

  7) Deinterlace — bwdif mode=send_field parity=auto
     Bob Weaver Deinterlacing Filter — calitate superioara yadif.
     Reconstruieste frame-uri complete din campuri interlasate.
     parity=auto: detecteaza automat TFF/BFF din metadate.
     OBLIGATORIU pt: DVD-uri rip, capturen TV analogic/digital,
     VHS digitizat, orice sursa interlasata (linii orizontale vizibile
     in miscare rapida).
     NU folosi pe surse progressive (camera, drone, telefon) —
     bwdif pe progressive genereaza artefacte.
     Impact performanta: mediu.

  8) Custom — scrii filtrul manual
     Introdu orice filtru ffmpeg valid (ex: eq=brightness=0.05:contrast=1.1,
     hflip, vignette=PI/4, colorbalance=rs=0.1).
     Filtrul se insereaza direct in lantul -vf, combinat cu scale/fps.
     Nu se face validare — sintaxa incorecta opreste encode-ul.

  COMBINATII RECOMANDATE:
  ─────────────────────────────────────────────────────────────
  Sursa TV/DVD interlasata + denoise:
    → Alege deinterlace (7) si ruleaza a doua oara cu denoise medium (3)
      (doua filtre combinate automat daca amandoua sunt in meniu —
       varianta viitoare; deocamdata alege custom si scrie manual:
       bwdif=mode=send_field:parity=auto,hqdn3d=4:3:6:4.5)

  4K drone cu ISO ridicat:
    → Denoise light (2) — pastreaza detalii, elimina granulatie

  Vechi digitizat / VHS:
    → Denoise strong (4) — curata agresiv artefactele de compresie/analogic

  AUDIO OUTPUT
  ────────────
  1) AAC 192k / 5.1 384k / 7.1 768k [implicit] — lossy, compatibilitate maxima
  2) AAC custom — alegi bitrate manual (ex: 128k, 256k, 320k)
  3) Opus 128k / 5.1 256k / 7.1 512k — lossy, calitate superioara la bitrate mic
  4) Opus custom — alegi bitrate manual
  5) FLAC lossless — zero pierdere calitate, fisier mai mare
  6) FLAC custom — alegi compression level (0-12, 8=default)
  7) E-AC3 (Dolby Digital Plus) — stereo 224k / 5.1 640k / 7.1 1024k
     Codec ffmpeg: -c:a eac3. Compatibil nativ cu Samsung TV + soundbar.
     Ideal pentru Home Cinema (TV recunoaste E-AC3 de pe stick USB).
     NOTA: ffmpeg NU suporta encodare Dolby Atmos JOC (Joint Object Coding).
     Conversia TrueHD → E-AC3 produce sunet surround de calitate, dar pierde
     obiectele spatiale Atmos. Dolby Encoding Engine (DEE) e singurul tool
     care face E-AC3+Atmos corect, dar e extern si comercial.
  8) LPCM (PCM necomprimat) — 16bit / 24bit / 32bit
     Codec ffmpeg: -c:a pcm_s16le / pcm_s24le / pcm_s32le
     Audio necomprimat, calitate maxima, fisier foarte mare.
     Ideal pentru mastering, editare profesionala, DaVinci Resolve.
     Compatibil cu mkv si mov. NU e compatibil cu mp4.
  9) Pastreaza audio original (copy) — fara re-encode

  Compatibilitate container (verificat din ffmpeg 8.1 Termux build):
    mp4: AAC, Opus (ffmpeg >= 4.3), E-AC3
    mkv: AAC, Opus, FLAC, E-AC3, LPCM — toate
    mov: AAC, LPCM (FLAC, E-AC3, Opus incompatibile)
    mxf: LPCM (celelalte incompatibile)
  Daca alegi FLAC + mp4/mov, E-AC3 + mov, LPCM + mp4 sau WebM + non-Opus,
  scriptul ofera switch automat.

  COMPATIBILITATE CONTAINER VIDEO + AUDIO (date reale ffmpeg 8.1):
  ───────────────────────────────────────────────────────────────
  VIDEO ENCODERE (ce putem SCRIE):
    libx265 HEVC  → mp4, mkv, mov
    libx264 AVC   → mp4, mkv, mov
    libsvtav1     → mp4, mkv, webm
    libaom-av1    → mp4, mkv, webm
    dnxhd DNxHR   → mkv, mov, mxf
    prores_ks     → mov (OBLIGATORIU)
    libopenapv    → NU DISPONIBIL in build-ul standard Termux
                    (necesita --enable-libopenapv la compilare)

  VIDEO DECODERE (ce putem CITI):
    H.264, H.265/HEVC, AV1, ProRes, ProRes RAW, DNxHR, APV
    MPEG-2, MPEG-4, VP9 (decodere native)
    Input: mp4, mkv, mov, m2ts, mts, vob, mxf, apv (raw)

  AUDIO ENCODERE (ce putem SCRIE):
    AAC            → mp4, mkv, mov
    Opus           → mp4 (ffmpeg 4.3+), mkv
    FLAC           → mkv
    E-AC3          → mp4, mkv
    LPCM pcm_s*    → mkv, mov, mxf
    Copy           → orice container compatibil cu sursa

  COMBINATII VALIDE (output):
    x265/x264 + AAC/Opus/E-AC3      → mp4, mkv
    x265/x264 + AAC                  → mov
    x265/x264 + FLAC/LPCM           → mkv
    AV1 + AAC/Opus/E-AC3            → mp4, mkv
    AV1 + Opus                       → mp4, mkv, webm
    AV1 + FLAC/LPCM                 → mkv
    DNxHR + AAC/LPCM                → mov
    DNxHR + orice                   → mkv, mxf
    ProRes + AAC/LPCM               → mov (OBLIGATORIU)
    APV — NU SE POATE ENCODA (libopenapv lipseste din build)
    Nota: WebM suporta DOAR Opus ca audio. Daca alegi alt codec
    cu WebM, scriptul ofera switch automat.

  NOTA DESPRE APV:
    Samsung APV (Advanced Professional Video) — codec intra-frame profesional.
    Dezvoltat de Samsung, open-source prin AcademySoftwareFoundation/openapv.
    Suporta: 422/444 10-bit si 12-bit, metadata HDR10/HDR10+, multi-view.
    Optimizat ARM NEON + x86 SSE/AVX. Container: mp4/mov (APV-ISOBMFF).

    ENCODARE APV (av_encoder_apv.sh):
      Necesita ffmpeg compilat cu --enable-libopenapv.
      Build-ul standard Termux NU include libopenapv.
      Scriptul are runtime check — afiseaza eroare clara daca lipseste.
      Pentru encodare APV: recompileaza ffmpeg din sursa cu flag-ul.

    DECODARE APV (citire fisiere APV):
      Decoderul APV ESTE inclus in ffmpeg 8.1 standard.
      Fisierele APV pot fi citite si convertite la orice alt codec.
      Exemplu: APV HDR10+ → x265 HDR10+ (cu metadata, ~90% reducere spatiu)
      sau APV → AV1 (si mai mic, dar metadata HDR10+ doar cu libsvtav1).
      NOTA: hdr10plus_tool nu suporta extragere metadata din APV direct.
      Metadata HDR10+ din sursa APV se pastreaza DOAR daca ffmpeg o transfera
      la re-encode (comportament ffmpeg standard pentru color metadata).

  Surround detection:
    Daca sursa are >2 canale (5.1), bitrate-ul creste automat:
      AAC 192k → 384k, Opus 128k → 256k, E-AC3 224k → 640k
    Daca sursa are >6 canale (7.1), bitrate-ul creste la:
      AAC 192k → 768k, Opus 128k → 512k, E-AC3 224k → 1024k
    La custom, bitrate-ul ramine cel introdus de utilizator.

  Avertizari metadata audio (per fisier, in log):
    TrueHD + re-encode: avertizare ca metadata Atmos (obiecte spatiale JOC)
    se va pierde — ffmpeg nu suporta encodare Atmos.
    DTS-HD MA / DTS:X + re-encode: avertizare ca metadata lossless/spatiala
    se va pierde — ffmpeg nu poate encoda DTS-HD (format proprietar Xperi).
    Avertizarile NU blocheaza encodarea, doar informeaza utilizatorul.

  Primul track audio se re-encodeaza, restul se copiaza intact (-c:a copy).

  MOD ENCODARE
  ────────────
  CRF   — Calitate constanta. Scriptul alege automat per rezolutie:
            x265: 4K=22, 1080p=21, 720p=20
            x264: 4K=20, 1080p=19, 720p=18
            AV1:  4K=30, 1080p=28, 720p=26
          Sau alegi CRF custom (0 = lossless, valori mai mari = mai mic/mai slab).
  VBR   — Bitrate mediu tinta (ex: 4000k, 4M). Maxrate = 1.5x, bufsize = 2x auto.
          Poti suprascrie maxrate/bufsize manual.

  PRESET
  ──────
  x265/x264: ultrafast → veryslow (9 niveluri). slow = recomandat telefon.
  SVT-AV1:   0 (veryslow) → 12 (ultrafast). 6 (medium) = recomandat.
  libaom:    cpu-used 0 (lent) → 8 (rapid). 4 = recomandat.
  Mai lent = calitate mai buna la acelasi bitrate, dar dureaza mai mult.

  TUNE / FILM-GRAIN
  ──────────────────
  x265/x264: animation, grain, film, stillimage, fastdecode, sau fara.
  AV1:       film-grain synthesis 0-50.
               0 = dezactivat.
               1-10 = usor (scena curata cu putin grain).
               11-20 = mediu (film 35mm tipic).
               21-50 = intens (grain vizibil, material vintage).
             SVT-AV1: integrat in -svtav1-params (film-grain=N).
             libaom: via -denoise-noise-level N.

  PARAMETRI EXTRA ENCODER (toate platformele)
  ──────────────────────────────────────────
  x265: format key=value separati cu ":" — ex: rc-lookahead=40:psy-rd=1.5
  x264: identic — ex: ref=6:bframes=5
  SVT-AV1: identic — ex: enable-overlays=1:scd=1
  libaom: format ffmpeg standard cu spatii — ex: -enable-chroma-deltaqp 1
  Pe toate platformele: Enter = sari (fara parametri extra).

8. DIALOGURI AUTOMATE (HDR / DV / DJI)
═══════════════════════════════════════════════════════════════

  Scripturile detecteaza automat situatii speciale si afiseaza dialog interactiv.

  DOLBY VISION DETECTAT (x265/x264)
  ──────────────────────────────────
  Afiseaza profil DV (4/5/7/8.x/9) si ofera:
    1) Stream copy video — pastreaza DV intact, re-encodeaza doar audio.
       Cel mai rapid. Fisierul ramane DV compatibil.
    2) Re-encode la HDR10 — pierde layer-ul DV, pastreaza HDR10 base.
       Util daca player-ul nu suporta DV.
    3) Sari fisierul.

  DOLBY VISION DETECTAT (AV1)
  ────────────────────────────
  AV1 nu suporta DV nativ. Optiuni:
    1) Converteste la HDR10 (pierde layer DV).
    2) Sari fisierul.

  HDR DETECTAT + X264 / DIALOG x264 PER-FILE
  ─────────────────────────────────────────────
  x264 nu suporta HDR nativ (fara metadata MaxCLL/MaxFALL/mastering).
  Dialog per-file cu analiza sursa:
    1) 8-bit (high — compatibilitate maxima)
    2) 10-bit (high10 — calitate, anti-banding)
    3) Stream copy video (pastreaza tot, rapid)
    4) Sari acest fisier
  Daca sursa e HDR/DV/HDR10+: warning suplimentar ca metadata se pierde.
  Daca sursa e LOG: redirecteaza la dialog LOG (fara optiune HDR10).

  FISIER DJI DETECTAT
  ────────────────────
  Afiseaza track-urile gasite (djmd, tmcd, dbgi, cover JPEG).
  Track-urile DJI (djmd/dbgi/tmcd) au codec "none" — incompatibil cu mp4/mov.
  Doar mkv poate pastra aceste track-uri.

  Daca container = mp4 sau mov:
    1) Schimba la MKV — pastreaza track-urile DJI (trece la dialogul MKV).
    2) Continua mp4/mov fara track-uri DJI (doar video + audio).
       Metadatele GPS/telemetrie raman in fisierul original.

  Daca container = mkv (sau dupa switch la MKV):
    Dialog selectie track-uri DJI in output:
    Cu dbgi prezent (4 optiuni, implicit = 2):
      1) Pastreaza tot (djmd + dbgi + tmcd)
      2) Fara debug (dbgi ~295 MB) [recomandat]
      3) Fara GPS/locatie (elimina djmd + dbgi, pastreaza tmcd)
      4) Elimina tot (fara track-uri DJI)
    Fara dbgi (3 optiuni, implicit = 1):
      1) Pastreaza tot (djmd + tmcd)
      2) Fara GPS/locatie (elimina djmd, pastreaza tmcd)
      3) Elimina tot (fara track-uri DJI)

  Cover JPEG (thumbnail DJI) nu se copiaza niciodata — este al doilea
  stream video si ar cauza re-encode MJPEG→libx265 (codec conflict).

  Containerul se schimba PER FISIER — daca ai un batch cu DJI + non-DJI,
  fiecare fisier primeste dialogul individual, iar containerul se reseteaza
  la cel original ales inainte de fiecare fisier.

  SUBTITRARI PGS/DVDSUB + CONTAINER MP4/MOV
  ───────────────────────────────────────────
  PGS si DVDSUB sunt formate imagine, incompatibile cu mov_text (mp4/mov).
  Scriptul le omite automat (-sn) si afiseaza avertisment.
  Solutie: foloseste mkv ca sa pastrezi subtitrari PGS/DVDSUB.

  ANALIZA SURSA — HDR10 (x265 / AV1)
  ────────────────────────────────────
  Cand sursa este HDR10 (fara HDR10+, fara DV, fara LOG), dialog per-file:
    1) Encodeaza HDR10 10-bit — pastreaza HDR10 cu color flags BT.2020/PQ.
    2) Encodeaza SDR 10-bit (tonemap Rec.709) — conversie la SDR prin
       linearizare + tonemap Hable + remap bt709. Util pentru ecrane non-HDR.
    3) Stream copy video — pastreaza tot, rapid.
    4) Sari acest fisier.

  ANALIZA SURSA — SDR (x265 / AV1)
  ──────────────────────────────────
  Cand sursa este SDR (8-bit sau 10-bit), dialog per-file:
    1) Encodeaza 10-bit SDR — output yuv420p10le (anti-banding, eficienta).
    2) Stream copy video.
    3) Sari acest fisier.

  Nota: x265 si AV1 encodeaza mereu in 10-bit (eficienta superioara).
  x264 are dialog-ul propriu cu alegere 8-bit / 10-bit / copy / skip.

  LOG FORMAT VIDEO DETECTAT (Apple Log / D-Log M / Samsung Log)
  ──────────────────────────────────────────────────────────────
  Detectie automata surse LOG de la iPhone (Apple Log), DJI Action/Pocket
  (D-Log M), Samsung S24 Ultra (Samsung Log). Se activeaza cand sursa
  este 10-bit + BT.2020 + brand detectat din metadata ffprobe.

  Meniu per-file pentru x265/AV1:
    1) Apply LUT (.cube) → 10-bit SDR Rec.709
       Apare DOAR daca un fisier .cube corespunzator exista.
    2) Convert SDR (fara LUT) → 10-bit Rec.709
       Tonemap best-effort cu zscale (linearizare + Hable + bt709).
       LUT-ul oficial este recomandat pentru acuratete maxima.
    3) Convert HDR10 (fara LUT) → 10-bit BT.2020/PQ (HDR10 static)
       x265: x265-params hdr10=1 | AV1: color flags directe.
    4) Preserve Log — compresie fara schimbare culori
       Pastreaza color flags originale. CRF recomandat: 12-15.
    5) Stream copy video
    6) Sari fisier

  Meniu per-file pentru x264:
    Fara optiune HDR10 (x264 nu suporta).
    Forteaza 8-bit (High profile). Warning la Preserve Log.

  DNxHR / ProRes:
    Nu afiseaza dialog — pastreaza automat profilul Log intact.
    Mesaj informativ in log.

  CAUTARE LUT (.cube)
  ────────────────────
  Scriptul cauta automat LUT-uri intr-o singura locatie per platforma:

    Windows (PS1): $InputDir/Luts/      (langa script, case-insensitive NTFS)
                   $InputDir/Luts/Creative/  (LUT-uri creative/artistice)

    Android (SH):  /storage/emulated/0/Media/Luts/
                   /storage/emulated/0/Media/Luts/Creative/

  Nu mai scaneaza alte foldere — un singur loc, clar, fara ambiguitate.

  Conventia de denumire (prefix obligatoriu):
    apple_log_*.cube     — ex: apple_log_AppleLogToRec709-v1.0.cube
    dji_dlog_m_*.cube    — ex: dji_dlog_m_Rec709.cube
    samsung_log_*.cube   — ex: samsung_log_Rec709_3DLUT_v1.0.cube

  Daca un singur LUT e gasit → selectie automata.
  Daca mai multe LUT-uri → dialog de selectie.
  Daca brand-ul e necunoscut → listeaza toate .cube din folder.

  Surse oficiale LUT-uri:
    Apple:   developer.apple.com → "Apple Log Profile Delivery SDK"
    DJI:     dji.com → pagina produsului → Downloads
    Samsung: developer.samsung.com/mobile/samsung-log-video.html
             Foloseste: Samsung Log to Rec709 3DLUT v1.0

  FORCE LOG DETECTION
  ────────────────────

    1) Auto-detect [implicit] — detectie pe baza metadata ffprobe
    2) Fortat ON — trateaza toate fisierele ca LOG

  Util cand camera nu seteaza metadata corect (camere chinezesti,

  Se salveaza in profile .conf.

  VFR (Variable Frame Rate) WARNING
  ──────────────────────────────────
  Cand sursa LOG vine de pe telefon (iPhone, Samsung), este posibil
  sa fie VFR (Variable Frame Rate). Scriptul detecteaza automat
  diferenta intre avg_frame_rate si r_frame_rate si afiseaza warning
  in dialog-ul LOG cu recomandarea de a seta FPS fix din meniu.

  HDR10+ DETECTAT (x265 / AV1)
  ─────────────────────────────
  Cand sursa contine metadata HDR10+ dinamica, dialog per-file:

  Cu hdr10plus_tool instalat:
    1) Re-encode HDR10 static — pierde metadata dinamica (+), pastreaza HDR10 base.
    2) Re-encode HDR10+ — extrage metadata JSON via hdr10plus_tool si o
       reinjecteaza in output (x265: dhdr10-info, AV1/SVT: hdr10plus-json).
    3) Stream copy video — pastreaza tot, rapid.
    4) Triple-layer DV+HDR10+HDR10+ [doar daca si dovi_tool instalat]
       Pipeline: extract HDR10+ JSON → genereaza DV RPU (Profile 8.1 CMv4.0)
       → encode x265 cu dhdr10-info → inject DV RPU → re-mux.
       Rezultat: fisier cu 3 layere — DV (compatibil DV TV), HDR10 (fallback),
       HDR10+ (metadata dinamica). Cel mai compatibil output posibil.

  Fara hdr10plus_tool:
    1) Re-encode HDR10 static — pierde metadata dinamica.
    2) Stream copy video.

  INTERACTIVE MODE (dupa fiecare fisier)
  ──────────────────────────────────────
  Cand Interactive Mode e activat (setare din meniu sau profil .conf),
  dupa fiecare fisier encodat cu succes apare dialog:
    1) Pastreaza setarile (continua) [implicit]
    2) Modifica setari — audio, CRF, normalizare
    3) Opreste batch-ul
  Util pentru batch-uri mixte (ex: unele fisiere necesita alt audio sau CRF).

  PRORES ENCODER (v32)
  ─────────────────────
  Encoder Apple profesional intra-frame (prores_ks).
  Container obligatoriu: .mov (QuickTime).
  Profile disponibile:
    Proxy    — ~45 Mbps (offline editing cu spatiu limitat)
    LT       — ~100 Mbps (light workflow)
    Standard — ~145 Mbps (interchange standard)
    HQ       — ~220 Mbps [implicit recomandat] (high quality, broadcast)
    4444     — ~330 Mbps (4:4:4, alpha channel support)
    4444 XQ  — ~500 Mbps (calitate maxima, VFX/compositing)
  LOG format: ProRes pastreaza profilul Log intact automat (fara dialog).
  Track-uri DJI: omise automat (mov incompatibil cu codec 'none').

  WEBM CONTAINER PENTRU AV1 (v32)
  ────────────────────────────────
  AV1 suporta container WebM (nativ web, VP8/VP9/AV1).
  WebM suporta DOAR audio Opus/Vorbis — scriptul avertizeaza automat
  daca audio-ul ales nu e Opus si ofera conversie sau switch la MKV.
  Subtitrari: omise automat (-sn) — WebM suporta doar WebVTT.


  HARDWARE ENCODE GPU (v35 overhaul, PS1 only)
  ─────────────────────────────────────────────
  Meniu encoder optiunea 6. Detectie dubla: ffmpeg binary + GPU fizic (WMI).

  NVIDIA NVENC: hevc_nvenc, h264_nvenc, av1_nvenc
  Intel QSV:    hevc_qsv,   h264_qsv,   av1_qsv
  AMD AMF:      hevc_amf,   h264_amf,   av1_amf (nou in v35)

  Foloseste QP (Quality Parameter) in loc de CRF — recomandat 18-28.
  Preset-uri: NVENC p1-p7, QSV veryfast-veryslow, AMF speed/balanced/quality.
  10-bit automat (p010le) pe NVENC/QSV si AV1 AMF daca sursa e 10-bit.
  Avantaj: encode 10-20x mai rapid decat software pe GPU dedicat.
  Dezavantaj: calitate usor inferioara la acelasi bitrate vs software.

  Cerinte AV1 HW encode (generatii specifice):
    NVIDIA: RTX 40 series (Ada) sau mai nou (RTX 50, L4, L40)
    Intel:  Arc A/B series, Core Ultra (Meteor/Arrow/Lunar Lake)
    AMD:    RDNA3+ (RX 7000/8000/9000, W7000, iGPU 740M/760M/780M/860M/880M/890M)

  BANNER GPU la startul fluxului encode:
    Afiseaza toate GPU-urile fizice detectate (filtrand RDP/VM/virtual)
    Listeaza HW encode disponibil per-codec H.264/H.265/AV1
    Daca nu e detectat niciun GPU fizic: mesaj "HW indisponibil, doar SW"

  FILTRARE MENIU HW:
    Apar doar optiuni compatibile cu hardware-ul curent.
    Optiunile indisponibile apar in lista gri cu motiv clar:
    "Intel QSV AV1 (necesita Arc/Core Ultra)"
    "AMD AMF AV1  (necesita RDNA3+ RX 7000)"
    "AMD AMF *    (GPU AMD absent)"

  PROFILE .conf — validare HW la load:
    HW_ENC_CODEC salvat in profil se verifica impotriva GPU-ului real.
    Daca GPU nu suporta → fallback automat la SW echivalent:
      hevc_* → libx265    h264_* → libx264    av1_*  → libsvtav1
    Bypass: seteaza HW_FORCE=1 in profil (util pt GPU-uri noi
    ne-detectate de regex-ul intern — ex: viitoare generatii nesuportate).


  HARDWARE ENCODE MEDIACODEC (v38, Termux/Android only)
  ──────────────────────────────────────────────────────
  Meniu HW MediaCodec aparut in launcher dupa selectia encoder-ului
  (x265/x264/av1). Detectie dubla: ffmpeg binary (h264/hevc/av1_mediacodec)
  + Termux + getprop.

  ENCODERE MEDIACODEC:
    hevc_mediacodec — H.265 HW (8-bit + 10-bit pe SoC verificat)
    h264_mediacodec — H.264 HW (8-bit only, SDR only)
    av1_mediacodec  — AV1 HW (rar, doar SoC AV1-capable)

  SoC WHITELIST (capabilitati verificate, marcat [verificat] in meniu):
    Qualcomm Snapdragon 8xx (8 Gen 1+) — HEVC 10-bit
                            (8 Gen 2+) — AV1 encode
    Samsung Exynos 21xx-24xx           — HEVC 10-bit
                   24xx+               — AV1 encode
    Google Tensor G2+                  — HEVC 10-bit
                   G3+                 — AV1 encode
    MediaTek Dimensity 9000+           — HEVC 10-bit
                       9300+           — AV1 encode

  SoC NEDETECTAT IN WHITELIST: marcat [suport necunoscut] dar selectabil.
  Prompt confirmare la alegere (default: NU). Bypass: HW_FORCE=1 in profil.

  PIERDERI REALE (universale MediaCodec, nu chip-specific):
    HDR10+ dynamic metadata (per-frame) — imposibil via API public Android
    Dolby Vision native preservation    — necesita licenta Dolby (libdovi privat)

  TOT RESTUL FUNCTIONEAZA:
    HDR10 static (cu signaling repair via hevc_metadata bsf post-encode)
    HEVC 10-bit (main10) pe SoC verificat
    AV1 SDR pe SoC AV1-capable
    H.264 SDR oricare SoC

  DIALOG HDR UNIFORMIZAT (per fisier HDR cand MediaCodec selectat):
    Surse DV / HDR10+ / HDR10 → meniu cu 4-5 optiuni:
      [1] SW libx265 — pastrare completa (recomandat, default)
      [2] SW libx265 — strip enhancement (DV→HDR10 BL, HDR10+→HDR10 static)
      [3] MediaCodec — HDR10 10-bit + signaling repair (mastering display
          + max_cll/max_fall extrase din sursa via ffprobe, fallback Rec.2020
          1000/400 nits daca lipsesc)
      [4] MediaCodec — SDR tonemap 8-bit (proxy rapid pentru editare/preview)
      [5] Skip fisier
    Default mereu pe optiunea SW (apasa Enter pentru calitate maxima).

  H.264 SPECIAL CASE:
    h264_mediacodec NU suporta HDR — pe surse HDR forteaza tonemap automat
    sau fallback SW. Optiunea [3] HDR10 repair se comuta automat la [4] SDR.

  AV1 PRE-CHECK:
    Daca MC_CAP_AV1=0 (SoC fara AV1 HW encoder) → fallback automat la
    SW (libsvtav1/libaom-av1) cu mesaj informativ. Util sa selectezi
    MediaCodec pe AV1 fara sa stii cu siguranta daca SoC-ul are AV1 encoder.

  RATE CONTROL:
    MediaCodec NU suporta CRF — doar VBR/CBR via -b:v.
    get_adaptive_bitrate mapeaza CRF echivalent la kbps per codec×rezolutie:
      4K HEVC: 25 Mbps   1080p HEVC: 8 Mbps    720p HEVC: 4.5 Mbps
      4K H264: 35 Mbps   1080p H264: 12 Mbps   720p H264: 6.5 Mbps
      4K AV1:  18 Mbps   1080p AV1:  5.5 Mbps  720p AV1:  3 Mbps
    Maxrate = 1.5x bitrate, bufsize = 2x bitrate.

  PROFILE FIELD MEDIACODEC_HDR_POLICY (.conf bypass dialog pe batch):
    sw_full      — fallback SW completa
    sw_degraded  — fallback SW degradata (strip DV/dynamic)
    hw_repair    — MediaCodec HDR10 + repair
    hw_sdr       — MediaCodec SDR tonemap 8-bit
    skip         — skip fisierele HDR
    gol          — afiseaza dialog interactiv per fisier (default)

  CAZ REAL S24 ULTRA (Snapdragon 8 Gen 3 / Exynos 2400):
    HEVC 8-bit + 10-bit:  YES
    HDR10 (cu repair):    YES
    HDR10+:               drop dynamic, ramane HDR10
    Dolby Vision:         imposibil — comuta la SW sau strip→HDR10 BL
    AV1 encode:           YES
    H.264:                YES (8-bit only)

  PS1 NU este atins: MediaCodec e Android only. Pe Windows ramane
  NVENC/QSV/AMF.

  HLG SUPPORT (v39, end-to-end)
  ─────────────────────────────
  HLG (BT.2100 HLG, transfer=arib-std-b67) este al 4-lea tip de sursa HDR
  cu dialog dedicat, alaturi de HDR10/HDR10+/Dolby Vision.

  DETECTIE: detect_source_info / Get-SourceInfo seteaza IS_HLG / isHLG
  cand color_transfer=arib-std-b67. Mutual-exclusive cu HDR10/HDR10+/DV/LOG
  (Apple Log poate raporta arib-std-b67, deci HLG ramane 0 cand LOG e detectat).

  DIALOG HLG (5 optiuni — bash handle_hlg_dialog / PS Show-HLGDialog):
    [1] HLG nativ — pastreaza signaling BT.2100 HLG (recomandat)
    [2] HLG → HDR10 (PQ) — zscale linearization + smpte2084
    [3] HLG → SDR (Rec.709) — zscale + tonemap=hable
    [4] Stream copy video — pastreaza tot
    [5] Skip

  PARAMS HLG x265 (NU foloseste hdr10=1):
    -pix_fmt yuv420p10le
    -color_primaries bt2020 -color_trc arib-std-b67 -colorspace bt2020nc
    -x265-params hdr-opt=1:repeat-headers=1:colorprim=bt2020:transfer=arib-std-b67:colormatrix=bt2020nc

  HW WINDOWS: NVENC/QSV/AV1-AMF suporta p010le + signaling HLG.
    h264_amf/hevc_amf raman 8-bit (signaling se transmite oricum).
    h264 HW: HLG → HDR10 auto-fallback la HLG nativ (no SEI HDR10 pe H.264).

  MEDIACODEC HLG (Termux): HLG nativ avantaj — signaling in transfer chars,
    fara nevoie de SEI repair (spre deosebire de HDR10 unde mastering display
    SEI trebuie reinjectat).
    Dialog MediaCodec HLG (5 optiuni):
      [1] MediaCodec — HLG nativ 10-bit (recomandat)
      [2] SW libx265 — HLG nativ 10-bit
      [3] MediaCodec — HLG → HDR10 (PQ) + signaling repair
      [4] MediaCodec — SDR tonemap 8-bit
      [5] Skip
    h264_mediacodec NU suporta HLG → auto-comut pe SDR tonemap.
    AV1 MediaCodec accepta HLG dacă SoC suporta (MC_CAP_AV1=1).

  LOG → HLG via LUT (Convenție Luts/hlg_*.cube):
    hlg_apple_log_*.cube — Apple Log → HLG
    hlg_dji_dlog_m_*.cube — D-Log M → HLG
    hlg_samsung_log_*.cube — Samsung Log → HLG
    hlg_*.cube — prefix generic (brand necunoscut)
  LUT-urile se cauta in $InputDir/Luts/ paralel cu cele Rec.709.
  Dialog LOG (x265/AV1 only) primeste 2 optiuni noi:
    "Apply LUT Log→HLG → 10-bit BT.2100" (cand fisier hlg_*.cube gasit)
    "Convert HLG (fara LUT) → 10-bit" (zscale best-effort)

  PIPELINE/TRIM-CONCAT HLG: detect_pipeline_hdr_mode returneaza "hlg" cand
  toate input files sunt HLG. Pass 3 inject params HLG corect (fara hdr10=1).

  PROFILE FIELD: MEDIACODEC_HDR_POLICY=hw_hlg pentru bypass dialog HLG MediaCodec.

  PASTRARE STRUCTURA FOLDERE (v33.3)
  ──────────────────────────────────
  La inceputul batch-ului (dupa dry-run), intrebare noua:
    "Pastrezi structura de foldere din input? (d/n)"

  Daca n (implicit) — comportament anterior:
    - Toate fisierele din Input → acelasi folder Output
    - Scanare doar in radacina Input (fara subfoldere)

  Daca d — structura pastrată:
    - Scanare recursiva a tuturor subfolderelor din Input
    - Structura subfoldere recreata in Output
    - mkdir -p (bash) / New-Item -ItemType Directory (PS1)

  Exemplu:
    Input/Airsoft_20260405/clip1.mp4 → Output/Airsoft_20260405/clip1_x265.mp4
    Input/Moto_20260406/clip2.mp4   → Output/Moto_20260406/clip2_x265.mp4

  La scanare afiseaza:
    "Gasite: 25 fisiere in 4 foldere"

  La batch summary final:
    "Structura foldere: PASTRATA (4 foldere)"

  Disponibil pe Termux (SH) si Windows (PS1).

10b. TELEMETRY UNIFIED (v40) — DJI/GoPro/Sony/Garmin/QuickTime
═══════════════════════════════════════════════════════════════

  CE FACE:
    Extragere unificata telemetrie din video pentru 5 branduri majore.
    Brand auto-detectat per fisier (codec_tag + ISO 6709 fallback).
    Output normalizat cross-brand: <name>_norm.csv (schema 18 coloane).

  BRAND DETECTION (codec_tag scan):
    djmd | dbgi  → DJI         (ExifTool)
    gpmd         → GoPro       (parser GPMF Python)
    fdsc         → Garmin VIRB (parser FIT Python)
    nmea | sony  → Sony Action (parser NMEA Python)
    fallback     → QuickTime   (atom ISO 6709 via ExifTool, single-point)

  EXCLUDED v40: Insta360, Yi/Akaso/SJCAM (decizie utilizator).

  6 OPTIUNI MENIU:
    1) Standard       — GPX + CSV basic + CSV norm
    2) Full Data      — GPX + CSV FULL + CSV norm (toate metadatele disponibile)
    3) Subtitrare SRT — pentru overlay in VLC
    4) Totul          — GPX + CSV basic + CSV FULL + SRT + CSV norm
    5) Raw streams    — exporta track-ul nativ (DJI: djmd/dbgi/tmcd/cover;
                        GoPro: gpmd; Sony: nmea; Garmin: fdsc); QT: N/A
    6) Strip metadata — remux fara re-encode (sterge tracks)
                        DJI sub-dialog: 1)dbgi-only 2)djmd+dbgi 3)tot
                        Restul: orice optiune sterge track-ul telemetry

  CSV NORMALIZAT — schema 18 coloane (cross-brand):
    timestamp, lat, lon, alt_m, speed_mps, speed_kmh, heading_deg,
    gforce_x, gforce_y, gforce_z, gyro_x, gyro_y, gyro_z,
    temp_c, hr_bpm, cadence_rpm, power_w, source_brand

    source_brand valori:
      dji, gopro, sony, garmin, quicktime,
      external_gpx, external_fit, external_kml

  EXEMPLE:
    Termux:  cd $HOME && ./av_telemetry.sh   (sau prin av_launcher.sh opt 4)
    Windows: pwsh -File av_telemetry.ps1     (sau prin av_encode.ps1 opt 4)

    Pune fisierele video in InputVideos/, output in OutputVideos/output/.

  PARSERS DETALII:
    GPMF (GoPro): KLV binary, FourCC + 1B type + 1B struct_size + 2B BE sample_count.
                  Containere DEVC → STRM nested. SCAL aplicat la GPS5 (5×int32:
                  lat/lon/alt/speed_2d/speed_3d). Filter: skip GPSF<2 (no fix).
    FIT (Garmin): header validare ".FIT", definition+data messages.
                  mesg_num=20 record: lat/lon semicircles → degrees (×180/2^31),
                  alt=(raw/5)-500, speed=raw/1000. Extra: HR(3), cad(4),
                  power(7), temp(23). Timestamp: FIT epoch=1989-12-31.
    NMEA (Sony): $GPRMC + $GPGGA (+$GNRMC/$GNGGA multi-constellation).
                 ddmm.mmmm → decimal grade. Knots → m/s (×0.514444).
                 Altitudine din $GPGGA atasata la ultimul $GPRMC.
    QuickTime:   atom com.apple.quicktime.location.ISO6709, 1 punct (start).
                 Apple iPhone, Samsung, Pixel, OnePlus, Xiaomi (Android stock).

  BURN-IN HUD: planificat v41 (3 niveluri):
    - Extract only (v40 actual)
    - Embed lossless (atasare track GPX/CSV ca data stream)
    - Burn-in re-encode (overlay grafic + re-encode forced)

12. TROUBLESHOOTING
═══════════════════════════════════════════════════════════════

  PROBLEMA: "Permission denied" la rulare script
  SOLUTIE:  chmod +x av_launcher.sh (si celelalte .sh)

  PROBLEMA: "command not found: ffmpeg"
  SOLUTIE:  pkg install ffmpeg -y (Termux)
            sau adauga ffmpeg in PATH (Windows)

  PROBLEMA: PS1 nu se ruleaza ("execution of scripts is disabled")
  SOLUTIE:  Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
            (ruleaza o singura data in PowerShell ca Administrator)

  PROBLEMA: termux-wake-lock nu functioneaza
  SOLUTIE:  pkg install termux-api
            + instaleaza Termux:API din F-Droid (aplicatia separata)

  PROBLEMA: "Nu am gasit fisiere video"
  SOLUTIE:  Termux — pune fisierele in /storage/emulated/0/Media/InputVideos/
            Windows — pune fisierele in acelasi folder cu scriptul

  PROBLEMA: termux-setup-storage nu cere permisiune
  SOLUTIE:  Reinstaleaza Termux din F-Droid (versiunea Play Store nu suporta)

  PROBLEMA: Fisierele DJI pierd metadata GPS/telemetrie
  SOLUTIE:  Track-urile DJI (djmd/dbgi/tmcd) au codec "none" — incompatibile
            cu mp4/mov. Alege mkv, sau cand scriptul detecteaza DJI si ai
            ales mp4, accepta optiunea "Schimba la MKV" din dialog.
            Metadatele raman oricum in fisierul original.

  PROBLEMA: Subtitrari PGS/DVDSUB pierdute la mp4/mov
  SOLUTIE:  Normal — PGS/DVDSUB sunt incompatibile cu mp4/mov.
            Scripturile le omit automat (-sn) si afiseaza avertisment.
            Foloseste mkv pentru a pastra subtitrari PGS.

  PROBLEMA: Dolby Vision — cum pastrez DV complet?
  SOLUTIE:  Alege "Stream copy video" in dialogul DV.
            Video-ul nu se re-encodeaza, doar audio-ul.
            Pentru re-encode, DV se converteste la HDR10 (best-effort).

  PROBLEMA: "Export date GPS/DJI" nu porneste — "exiftool nu a fost gasit"
  SOLUTIE:  Termux: pkg install exiftool -y (sau cpan App::ExifTool)
            Windows: descarca exiftool.exe de pe https://exiftool.org/
            si pune-l in acelasi folder cu av_encode.ps1

  PROBLEMA: Export GPS genereaza fisiere goale
  SOLUTIE:  Fisierul video nu contine date GPS. Datele GPS se salveaza
            DOAR daca ai filmat cu telecomanda GPS Bluetooth DJI conectata
            sau cu aplicatia DJI Mimo pornita pe telefon in timpul filmarii.
            DJI Action 6 nu are senzor GPS intern.

DRY-RUN MODE — v23.1
═══════════════════════════════════════════════════════════════

  Dupa configurare, inainte de lansare, apare meniul:
    1) Encodeaza normal [implicit]
    2) Dry-run (doar analiza, fara encode)

  In mod dry-run, scriptul parcurge tot batch-ul si afiseaza
  per fisier ce ar face: encoder, container, sursa, HDR tip,
  resize, FPS, filtre, audio, loudnorm, estimare dimensiune.

  NU ruleaza ffmpeg — zero encodare. Dureaza cateva secunde
  pe tot batch-ul. Util pt verificare setari inainte de
  batch lung (50+ fisiere, ore de encodare).

  Disponibil pe toate platformele: bash, bat, ps1.

RESUME INTERRUPTED BATCH — v23.1
═══════════════════════════════════════════════════════════════

  Fiecare fisier encodat cu succes este marcat in
  batch_progress.log (in OutputDir). La reluare, fisierele
  deja procesate sunt sarite automat (pe langa skip >1MB
  existent pe output).

  La finalizarea cu succes a intregului batch,
  batch_progress.log se sterge automat.

  Daca batch-ul e intrerupt (Ctrl+C, crash, eroare):
    - batch_progress.log ramane pe disc
    - la urmatoarea rulare, fisierele OK sunt sarite
    - doar cele neprocesat se reiau

  Disponibil pe toate platformele: bash, bat, ps1.
